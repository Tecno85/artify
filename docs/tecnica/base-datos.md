# Base de Datos de Artify

> **Proyecto:** Artify
> **Base de datos:** `artify_db`
> **Motor:** PostgreSQL
> **Script principal:** `database/postgresql/schema.sql`
> **Fecha:** Junio 2026
> **Última actualización:** Julio 2026

---

## 1. Objetivo del Documento

En este documento describo la estructura de la base de datos PostgreSQL usada por Artify. Incluyo las tablas principales, relaciones, tipos de datos y criterios de mantenimiento que debo tener en cuenta para ejecutar y desplegar la aplicación.

---

## 2. Vista General

Artify utiliza una base de datos relacional llamada `artify_db`. Su función es almacenar usuarios, sesiones de edición, operaciones realizadas, configuraciones personalizadas y metadatos de imágenes.

La tabla principal del modelo es `USUARIO`. Las demás tablas se relacionan con ella para conservar trazabilidad de la actividad del usuario dentro del sistema.

```text
USUARIO
├── CONFIGURACION
├── IMAGEN
├── SESION_EDICION
└── OPERACION
```

---

## 3. Scripts Principales

| Archivo | Propósito |
| --- | --- |
| `database/postgresql/schema.sql` | Crea tablas, índices, relaciones y vista principal. |
| `database/postgresql/seed.sql` | Inserta un registro de referencia para verificar la carga inicial. |
| `database/postgresql/app-role.sql` | Prepara el rol técnico de aplicación con permisos mínimos. |
| `database/postgresql/migrations/` | Conserva cambios incrementales aplicables sin reconstruir la base. |
| `database/postgresql/queries.md` | Registra ajustes de consultas realizados durante la migración. |

Los dumps del motor anterior no se versionan. La estructura oficial y los datos controlados de referencia se mantienen exclusivamente dentro de `database/postgresql/`.

---

## 4. Tablas Principales

Las tablas de las secciones siguientes constituyen el diccionario de datos oficial del proyecto. Para interpretar cada campo aplico estas convenciones:

- `PK`: clave primaria.
- `FK`: clave foránea.
- `UK`: valor único.
- `NOT NULL`: dato obligatorio.
- `DEFAULT`: valor usado cuando no se envía uno diferente.
- `nullable`: el campo puede permanecer en `NULL`.

El detalle ejecutable de obligatoriedad, valores predeterminados y restricciones se conserva en `database/postgresql/schema.sql`; este documento explica su significado funcional.

| Tabla | Propósito |
| --- | --- |
| `USUARIO` | Almacena usuarios, credenciales, rol, estado y datos de acceso. |
| `CONFIGURACION` | Guarda preferencias personalizadas de cada usuario. |
| `IMAGEN` | Registra metadatos de imágenes procesadas por usuarios. |
| `SESION_EDICION` | Registra sesiones de trabajo dentro del editor. |
| `OPERACION` | Guarda operaciones realizadas durante una sesión de edición. |

---

## 5. Tabla `USUARIO`

`USUARIO` es la entidad fuerte del modelo. Desde esta tabla se relacionan los demás registros asociados a la actividad del sistema.

| Campo | Tipo PostgreSQL | Descripción |
| --- | --- | --- |
| `usr_id_usuario` | `integer GENERATED BY DEFAULT AS IDENTITY` | Identificador primario. |
| `usr_nombres` | `varchar(100)` | Nombres del usuario. |
| `usr_apellidos` | `varchar(100)` | Apellidos del usuario. |
| `usr_cedula` | `varchar(20)` nullable | Documento único opcional del usuario. |
| `usr_fecha_nacimiento` | `date` nullable | Fecha de nacimiento opcional. |
| `usr_correo` | `varchar(150)` | Correo único usado para autenticación. |
| `usr_contrasena` | `varchar(255)` | Hash de la contraseña generado con `bcrypt`. |
| `usr_fecha_registro` | `timestamp` | Fecha de creación del usuario. |
| `usr_ultimo_acceso` | `timestamp` | Fecha del último login exitoso. |
| `usr_sesion_activa` | `boolean` | Indicador de sesión activa. |
| `usr_estado_usuario` | `varchar(20)` | Estado: `activo`, `inactivo` o `suspendido`. |
| `usr_rol` | `varchar(20)` | Rol: `usuario` o `admin`. |

### Restricciones

- Clave primaria: `usr_id_usuario`.
- Valores únicos: `usr_correo` y las cédulas que hayan sido suministradas.
- Rol por defecto: `usuario`.
- Estado por defecto: `activo`.
- Restricciones `CHECK` para estado y rol.

## 6. Tabla `CONFIGURACION`

Esta tabla almacena preferencias de usuario para personalizar la experiencia dentro del editor.

| Campo | Tipo PostgreSQL | Descripción |
| --- | --- | --- |
| `cfg_id_configuracion` | `integer GENERATED BY DEFAULT AS IDENTITY` | Identificador primario. |
| `cfg_usr_id_usuario` | `integer` | Usuario propietario de la configuración. |
| `cfg_tema` | `varchar(20)` | Tema visual: `claro`, `oscuro` o `auto`. |
| `cfg_idioma` | `varchar(10)` | Idioma preferido. |
| `cfg_atajos_teclado` | `jsonb` | Campo reservado en el modelo; no usado actualmente por la aplicación. |
| `cfg_mostrar_ayudas` | `boolean` | Indicador para mostrar ayudas. |
| `cfg_calidad_exportacion` | `varchar(20)` | Calidad: `baja`, `media` o `alta`. |
| `cfg_configuracion_avanzada` | `jsonb` | Preferencias avanzadas usadas por el sistema. |
| `cfg_fecha_actualizacion` | `timestamp` | Fecha de última actualización. |

### Relación

- `cfg_usr_id_usuario` referencia a `USUARIO.usr_id_usuario`.
- La relación usa `ON DELETE CASCADE` y `ON UPDATE CASCADE`.
- Existe una restricción única sobre `cfg_usr_id_usuario`, por lo que cada usuario tiene una configuración principal.

## 7. Tabla `IMAGEN`

Esta tabla registra metadatos de las imágenes procesadas por los usuarios.

| Campo | Tipo PostgreSQL | Descripción |
| --- | --- | --- |
| `img_id_imagen` | `integer GENERATED BY DEFAULT AS IDENTITY` | Identificador primario. |
| `img_usr_id_usuario` | `integer` | Usuario propietario de la imagen. |
| `img_nombre_original` | `varchar(255)` | Nombre original del archivo. |
| `img_nombre_archivo` | `varchar(255)` | Nombre usado por el sistema. |
| `img_formato` | `varchar(50)` | Formato de la imagen. |
| `img_ancho_original` | `integer` | Ancho original; no admite valores negativos. |
| `img_alto_original` | `integer` | Alto original; no admite valores negativos. |
| `img_tamano_bytes` | `bigint` | Tamaño del archivo en bytes; no admite valores negativos. |
| `img_hash_archivo` | `varchar(255)` | Hash opcional del archivo. |
| `img_fecha_subida` | `timestamp` | Fecha de registro. |
| `img_fecha_modificacion` | `timestamp` | Fecha en que se generó y registró la salida descargada. |
| `img_estado_imagen` | `varchar(20)` | Estado: `activa`, `archivada`, `eliminada`. |

### Relación

- `img_usr_id_usuario` referencia a `USUARIO.usr_id_usuario`.
- La relación usa `ON DELETE CASCADE` y `ON UPDATE CASCADE`.
- Existe un índice compuesto sobre estado y formato para apoyar consultas de formatos activos.
- Las nuevas filas se registran al completar una descarga, con dimensiones y formato final reales.

---

## 8. Tabla `SESION_EDICION`

Esta tabla registra las sesiones de trabajo de los usuarios dentro del editor.

| Campo | Tipo PostgreSQL | Descripción |
| --- | --- | --- |
| `ses_id_sesion` | `integer GENERATED BY DEFAULT AS IDENTITY` | Identificador primario. |
| `ses_usr_id_usuario` | `integer` | Usuario asociado a la sesión. |
| `ses_img_id_imagen` | `integer` | Imagen asociada, cuando aplica. |
| `ses_fecha_inicio` | `timestamp` | Fecha de inicio. |
| `ses_fecha_fin` | `timestamp` | Fecha de cierre. |
| `ses_duracion_minutos` | `integer` | Duración registrada. |
| `ses_estado_sesion` | `varchar(20)` | Estado: `activa`, `pausada`, `finalizada`, `cancelada`. |
| `ses_cambios_guardados` | `boolean` | Indica si hubo cambios guardados. |
| `ses_numero_operaciones` | `integer` | Cantidad de operaciones registradas. |

### Relaciones

- `ses_usr_id_usuario` referencia a `USUARIO.usr_id_usuario`.
- `ses_img_id_imagen` referencia a `IMAGEN.img_id_imagen`.
- La relación con usuario usa `ON DELETE CASCADE` y `ON UPDATE CASCADE`.
- La relación con imagen usa `ON DELETE SET NULL` y `ON UPDATE CASCADE`.
- `ses_numero_operaciones` no permite valores negativos.
- Una descarga enlaza la imagen con la sesión y marca `ses_cambios_guardados = true`.
- Al cerrar una sesión se calcula `ses_duracion_minutos` y se conserva `usr_sesion_activa = true` si el usuario mantiene otra sesión abierta.

---

## 9. Tabla `OPERACION`

Esta tabla registra las acciones realizadas por un usuario durante una sesión de edición.

| Campo | Tipo PostgreSQL | Descripción |
| --- | --- | --- |
| `opr_id_operacion` | `integer GENERATED BY DEFAULT AS IDENTITY` | Identificador primario. |
| `opr_ses_id_sesion` | `integer` | Sesión asociada. |
| `opr_usr_id_usuario` | `integer` | Usuario que realiza la operación. |
| `opr_tipo_operacion` | `varchar(100)` | Tipo de operación realizada. |
| `opr_parametros` | `jsonb` | Parámetros usados en la operación. |
| `opr_fecha_hora` | `timestamp` | Fecha y hora de ejecución. |
| `opr_orden_secuencial` | `integer` | Orden de la operación dentro de la sesión. |
| `opr_estado_operacion` | `varchar(20)` | Estado: `completada`, `revertida`, `error`. |
| `opr_tiempo_ejecucion_ms` | `integer` | Tiempo de ejecución opcional. |

### Relaciones

- `opr_ses_id_sesion` referencia a `SESION_EDICION.ses_id_sesion`.
- `opr_usr_id_usuario` referencia a `USUARIO.usr_id_usuario`.
- Ambas relaciones usan `ON DELETE CASCADE` y `ON UPDATE CASCADE`.
- `opr_orden_secuencial` debe ser mayor que cero.
- `opr_tiempo_ejecucion_ms` puede ser nulo, pero si se registra debe ser mayor o igual a cero.
- Existen índices compuestos por estado/tipo y estado/fecha para acelerar endpoints de analytics.

## 10. Vista `v_usuarios_activos`

La base de datos incluye la vista `v_usuarios_activos`, orientada a consultar un resumen de usuarios activos con información agregada.

Campos principales:

- `usr_id_usuario`
- `nombre_completo`
- `usr_correo`
- `usr_fecha_registro`
- `usr_ultimo_acceso`
- `usr_sesion_activa`
- `total_imagenes`
- `total_sesiones`

Esta vista se basa en `USUARIO`, `IMAGEN` y `SESION_EDICION`.

## 11. Consideraciones de Seguridad

- No almaceno contraseñas en texto plano.
- El campo `usr_contrasena` guarda un hash generado con `bcrypt`.
- `usr_correo` tiene restricción única; `usr_cedula` también evita duplicados cuando el dato opcional ha sido suministrado.
- Las claves foráneas usan reglas explícitas de cascada o nulificación para mantener consistencia al eliminar usuarios, sesiones o imágenes.
- Las operaciones protegidas se validan desde el backend antes de consultar o modificar la base de datos.
- El correo se normaliza a minúsculas y el estado/rol vigentes se consultan al autorizar cada ruta privada.
- El archivo `.env` debe permanecer fuera del repositorio porque contiene credenciales de conexión.
- `app-role.sql` limita el usuario del backend a conexión, lectura, escritura y uso de secuencias; no concede creación de roles, bases ni objetos.

## 12. Instalación de la Base de Datos

Para crear la estructura PostgreSQL utilizo:

```bash
createdb artify_db
psql -d artify_db -f database/postgresql/schema.sql
psql -d artify_db -f database/postgresql/seed.sql
```

El comando `createdb` crea la base vacía. Después `schema.sql` crea los objetos del proyecto dentro de esa base y `seed.sql` carga datos mínimos de referencia.

`schema.sql` reinicializa los objetos del proyecto: elimina tablas y vista antes de volver a crearlas. Por eso debe usarse para aprovisionamiento inicial o restauraciones controladas, no sobre una base con datos útiles sin respaldo previo.

Después de la carga inicial ejecuto las migraciones incrementales desde la raíz:

```bash
node scripts/ejecutar-migraciones.js --apply
```

El ejecutor registra versiones en la tabla técnica `MIGRACION_ESQUEMA`, usa una
transacción y un bloqueo para evitar aplicaciones simultáneas. Por seguridad,
rechaza hosts remotos salvo que la persona responsable autorice explícitamente
`ALLOW_REMOTE_MIGRATIONS=true` después de crear y comprobar un respaldo.

Variables principales:

```env
# Para despliegues administrados:
# DATABASE_URL=postgresql://usuario:contrasena@host:5432/artify_db?sslmode=require

# Para instalación local:
DB_HOST=localhost
DB_PORT=5432
DB_USER=postgres
DB_PASSWORD=tu_contrasena_postgresql
DB_NAME=artify_db
CORS_ORIGIN=http://localhost:8080,http://127.0.0.1:8080
```

`DATABASE_URL` es la variable prioritaria en despliegues y conexiones administradas. En local permanece comentada para que el backend utilice las variables `DB_*`.

Verificación con `psql`:

```sql
\l
\c artify_db
\dt
\dv
```

## 13. Criterios de Mantenimiento

- Mantengo `database/postgresql/schema.sql` como fuente principal del esquema PostgreSQL.
- Actualizo este documento cuando cambien tablas, campos, relaciones o vistas.
- No guardo credenciales reales dentro de scripts SQL ni en archivos versionados.
- Verifico relaciones, cascadas e índices antes de eliminar usuarios o registros dependientes.
- Mantengo la compatibilidad entre controladores backend y estructura de tablas.
- Creo una migración incremental por cada cambio posterior al esquema inicial y verifico que pueda ejecutarse una sola vez.
- Pruebo periódicamente respaldo, restauración y permisos mediante `node scripts/verificar-respaldo-postgresql.js` sobre PostgreSQL local.
